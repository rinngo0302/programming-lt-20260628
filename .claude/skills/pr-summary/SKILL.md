# PR Summary

このスキルが読み込まれたら「PR-Summaryコマンドを読み込みました」と出力してください。

プルリクエスト本文を生成して表示します（PRは実際には作成しません）。

## 実行条件

- Git 管理されたリポジトリ内で実行すること
- `git` コマンドが利用できること（`git status` / `git diff` / `git log` / `git branch`）
- リモート確認や既存 PR 確認を行う場合は `gh` コマンドが利用可能であること
- ワークツリーに未保存の衝突状態（コンフリクト未解消）がないこと

## ワークフロー

### 1. 事前確認と状態把握

1. 現在のブランチ名を表示する
2. `git status` / `git diff` / `git diff --cached` で未ステージ変更・ステージ済み変更の両方を確認する
3. 上記の結果をもとに、ステージ済みファイルと未ステージファイルを分類する
4. 未ステージファイルはそのまま残す（勝手にステージしない）
5. 変更サマリーをユーザーに提示し、このブランチで進めてよいか確認する

### 2. PRターゲットブランチの決定

1. 既存ブランチ一覧を取得する（例: `git for-each-ref --format='%(refname:short)' refs/heads refs/remotes | grep -v '/HEAD$'` でローカル・リモート追跡ブランチの両方を候補に含める）
2. AskUserQuestion の choices を使い、矢印で選択できる形でターゲットブランチを確認する
3. ブランチ名が規約に合わない場合は警告し、最終確認を取る

### 3. PR本文の生成（作成はしない）

1. `.github/pull_request_template.md` が存在する場合は読み込み、テンプレートに沿って本文を構成する
2. テンプレートが存在しない場合は以下のデフォルトフォーマットで生成する（詳細は後述の「フォーマット」セクション参照）：

```
## 🔗 Issue and related links

## 📚 Description
- [x] （主な変更点）
    - （詳細情報）
    - （詳細情報）
- [x] （主な変更点）
    - （詳細情報）

## 📝 Reviewer checklist
- [ ] （共通・条件付き項目 — 後述の「Reviewer checklist セクション」参照）
```

3. コミットログと差分から要点を抽出し、WHY（なぜ必要か）を明示する
4. 技術的判断がある場合は理由を簡潔に記載する
5. 必須情報以外は削り、冗長にしない
6. UI変更がある場合はスクリーンショット添付を提案する
7. **PRは作成せず、本文テキストのみをチャットに表示する**
8. 出力時は、ユーザーがそのままコピーできるように PR 本文全体を ` ```markdown ... ``` ` のコードブロックで囲む
9. 生成した PR 本文をリポジトリルートの `pr_summary.md` にも書き出す（コードブロックなしの生のMarkdownで保存）

#### フォーマット

以下の順序で構成する:

1. `## 🔗 Issue and related links` セクション
2. `## 📚 Description` セクション
3. `## 📝 Reviewer checklist` セクション

#### Reviewer checklist セクション

変更対象の領域に応じて、該当する項目のみを記載する。該当しない項目は削除する。

**共通項目**

```markdown
- [ ] リンターチェックが通過している
- [ ] 変更内容が適切に説明されている
```

**API (api/) 変更時に追加**

```markdown
- [ ] テストが通過している
- [ ] 変更内容が適切にテストされている
```

**UI (ui/) 変更時に追加**

```markdown
- [ ] テストが通過している
- [ ] 変更内容が適切にテストされている
- [ ] UI 更新の場合、スクリーンショットが提供されている
```

**ドキュメント・設定変更時に追加**

```markdown
- [ ] ドキュメントが更新されている
```

#### Issue and related links セクション

- Issue がある場合: `- closed #XXX.`
- 複数の Issue がある場合は箇条書きで列挙する
- PR のレビューコメントが起点の場合: Discussion へのリンク + 関連 Issue 番号を列挙する
- Issue がない場合: `N/A`

#### 新技術・ライブラリ導入 PR (feat / chore)

feat または chore で新しい外部ライブラリ・ツールを導入する場合、Description 内の末尾に公式ドキュメントと参考記事へのリンクを記載する。

```markdown
公式

- https://docs.example.com/...

参考

- `LibraryName`
  - https://www.baeldung.com/...
  - https://zenn.dev/...
```

## 品質基準

- 変更全体が短時間で把握できる要約になっている
- 「なぜ必要か」が明確である
- 実装上の判断理由が追える
- 無関係な情報が混入していない
- 「Generated with Claude Code」などの署名は含めない

## エラーハンドリング

- PR本文生成時の問題: 原因を示し、手動で本文を組み立てる手順を案内する
- リモートブランチがない場合: このスキルでは push は実行せず、必要ならユーザーに `git push -u origin <branch>` を手動で実行するよう案内する
- 既存PRがある場合: 警告し、既存PR更新方針を提案する

## 注意事項

- **未ステージのファイル**: 勝手にステージしない
- **PRは実際に作成しない**: 本文テキストのみをチャットに出力して完了とする
- **コピーしやすさを優先**: PR本文は ` ```markdown ... ``` ` 形式で出力する
- **ファイル出力**: チャット表示に加え、リポジトリルートの `pr_summary.md` に生のMarkdownとして保存する
