# Code Review

このスキルが読み込まれたら「Code Reviewコマンドを読み込みました」と出力してください。

> **モデル推奨**: 重要なレビュー（本番PR・セキュリティ関連・大規模変更）の場合は `/model` で **Opus** に切り替えてから実行すること。

## レビュー対象の判定

以下の優先順位で対象を判定する：

1. PR番号が指定された場合（例：`#123`）→ `gh pr diff <番号>` でPRの差分を取得
2. それ以外の場合 → AskUserQuestion を使って「比較元のターゲットブランチを教えてください（例: main, develop）」と必ず確認し、`git diff <ターゲットブランチ>...HEAD` で差分を取得
3. ブランチ差分が取得できない場合 → `git diff --cached` と `git diff` を両方実行し、staged + unstagedの変更を取得
4. 差分が取得できない（変更なし）の場合 → 「レビュー対象の変更が見つかりませんでした」と伝えて終了

## レビュー手順

1. 上記の判定ルールに従い差分を取得する（PR番号指定がない場合は必ずAskUserQuestionでターゲットブランチを確認してから差分を取得する）
2. PRレビューの場合は、まず「このPRが何を変更するものか」を3〜6行で要約する
3. AskUserQuestionを使って「背景情報の説明も必要ですか？」と確認する
4. ユーザーが必要と答えた場合は、初心者にも分かる言葉で以下を説明する
   - このPRがなぜ必要なのか（解決したい課題）
   - このPRを入れるとプロジェクトにどう影響するか（利用者・運用・保守への影響）
   - 必要に応じてプロジェクト構成（主要ディレクトリ/責務/このPR変更箇所の位置づけ）
5. 以下の5観点すべてでコードをレビューする
6. 動作確認を実施し、確認結果を整理する
    - Webフロントエンドの変更が含まれる場合は Claude in Chrome を使って確認する
    - テストが存在する場合は関連テストを実行し、結果（成功/失敗と要点）を記録する
7. 指摘内容を踏まえて最終アクションを判定する（マージ可 / 要修正 / 再レビュー推奨）
8. 規定の出力形式で結果を日本語で出力する

## レビュー観点

| 観点 | 確認内容 |
|------|----------|
| **バグ・ロジックエラー** | null参照、境界値エラー、競合状態、誤ったロジックなど |
| **コードスタイル・可読性** | 命名規則、関数の長さ、重複コード、コメントの適切さ |
| **セキュリティ脆弱性** | SQLインジェクション、XSS、認証・認可の不備、機密情報のハードコードなど |
| **パフォーマンス** | 不要なループ、N+1問題、メモリリーク、非効率なアルゴリズムなど |
| **テスト観点** | 変更に対するテストの過不足、境界値/異常系のカバレッジ、既存テストへの影響など |

スタイルや軽微なフォーマットの指摘は最小限にし、本質的な問題を優先する。
各指摘には「なぜ問題か」と「想定される影響（再現条件・影響範囲）」を1行で必ず含める。

## 出力形式

```
## コードレビュー結果

### 🧾 PR概要
このPRの目的と主な変更点の要約（3〜6行）

### 🌍 背景と影響（必要な場合のみ）
- なぜ必要か: 解決したい課題
- プロジェクトへの影響: ユーザー体験・運用・保守への影響
- プロジェクト構成（必要な場合のみ）: 主要ディレクトリ/責務/このPR変更箇所の位置づけ

### 🧪 動作確認
- 実施内容: 何をどう確認したか
- テスト実行: 実行したテストコマンドと結果（テストがある場合は必須）
- 結果: OK / NG（NGの場合は再現手順付き）
- 備考: Webフロントエンド変更時は Claude in Chrome で確認

### 🔴 重大度：高
- **[観点] ファイル名:行番号**（確信度: 高/中/低）— 問題の説明と改善提案  
  根拠/影響: なぜ問題か、再現条件または影響範囲

### 🟡 重大度：中
- **[観点] ファイル名:行番号**（確信度: 高/中/低）— 問題の説明と改善提案  
  根拠/影響: なぜ問題か、再現条件または影響範囲

### 🟢 重大度：低
- **[観点] ファイル名:行番号**（確信度: 高/中/低）— 問題の説明と改善提案  
  根拠/影響: なぜ問題か、再現条件または影響範囲

### ✅ 総評
全体的な評価と主な改善ポイントのまとめ

### 🚦 最終アクション
- **マージ可**: 高重大度の問題がなく、修正必須事項がない
- **要修正**: 高重大度の問題が1件以上ある、または動作に影響する修正必須事項がある
- **再レビュー推奨**: 修正後の影響範囲が広く、再確認が必要
```

各重大度に指摘事項がない場合は「指摘なし」と明記する。
